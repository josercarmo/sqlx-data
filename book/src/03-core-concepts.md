# Chapter 3: Core Concepts

Now that you've seen SQLx-Data in action, let's understand the fundamental concepts that make it work. This chapter focuses on the core building blocks you need to know before diving into advanced features.

## Code Convention Note

Throughout this book, you'll see example functions using the `&impl TraitName` pattern:

```rust
async fn create_multiple_users(repo: &impl BatchRepo) -> Result<()> {
    // Function body
}
```

**Important**: This signature style is **optional** and used only as examples to demonstrate usage patterns. In your actual implementation, you can use any function signature that suits your architecture:

- `async fn create_users(repo: MyRepo) -> Result<()>`
- `async fn create_users<T: BatchRepo>(repo: &T) -> Result<()>`
- Methods within your own structs and implementations
- Any other pattern that fits your design

The `&impl TraitName` pattern is simply a convenient way to show how the repository traits can be used generically.

## How the Macros Actually Work

### Macro Transformation Process

The SQLx-Data macros don't add methods to existing traits. Instead, they **completely replace** empty method signatures with full implementations.

#### What You Write:
```rust
#[repo]
trait UserRepo {
    #[dml("SELECT 1")]
    async fn ping(&self) -> Result<i64>;  // NO IMPLEMENTATION
}
```

#### What Gets Generated:
```rust
trait UserRepo {
    // Generated by #[dml] macro:
    async fn ping_query(&self) -> Result<i64> {
        sqlx::query_scalar!("SELECT 1").fetch_one(self.get_pool()).await
    }

    // The original method NOW HAS a default implementation
    async fn ping(&self) -> Result<i64> {
        self.ping_query().await
    }

    // Method added by the #[repo] macro
    fn get_pool(&self) -> &Pool {
        unimplemented!("Implement get_pool() to use methods without pool parameters")
    }
}
```

## The Repository Pattern Foundation

### What SQLx-Data Does

SQLx-Data implements the Repository Pattern through two key macros:

- **`#[repo]`** - Processes traits, adds `get_pool()` method, handles aliases and scopes
- **`#[dml]`** - Replaces empty method signatures with full SQLx implementations

```rust
#[repo]                                    // <- Enables trait processing
trait UserRepo {
    #[dml("SELECT * FROM users WHERE id = ?")]   // <- Replaces signature with implementation
    async fn find_by_id(&self, id: i64) -> Result<Option<User>>;
}
```

### The Implementation Contract

Every repository requires one thing: implement `get_pool()`:

```rust
struct App {
    pool: Pool,
}

impl UserRepo for App {
    fn get_pool(&self) -> &Pool {    // <- This is the only required implementation
        &self.pool
    }
    // All query methods are inherited automatically from the trait!
}
```

## Two-Method Generation Pattern

Each `#[dml]` method generates **two methods**:

### 1. The `_query` Method (Does the Real Work)
```rust
// Generated by #[dml("SELECT name FROM users WHERE id = ?")]
async fn get_name_query(&self, id: i64) -> Result<String> {
    sqlx::query_scalar!("SELECT name FROM users WHERE id = ?", id)
        .fetch_one(self.get_pool())
        .await
}
```

### 2. The Original Method (Default Implementation)
```rust
// Default implementation that calls the _query method
async fn get_name(&self, id: i64) -> Result<String> {
    self.get_name_query(id).await
}
```

This pattern allows you to:
- **Use the default**: Call `repo.get_name(id)`
- **Override with custom logic**: Implement `get_name()` but still use `get_name_query()`
- **Add logging/caching**: Wrap the `_query` method with additional functionality

## Return Type Intelligence

SQLx-Data analyzes your return type to choose the correct SQLx method:

### Single Results
```rust
#[repo]
trait UserRepo {
    // Result<T> -> uses fetch_one() - expects exactly one row
    #[dml("SELECT name FROM users WHERE id = ?")]
    async fn get_name(&self, id: i64) -> Result<String>;

    // Result<Option<T>> -> uses fetch_optional() - zero or one row
    #[dml("SELECT name FROM users WHERE id = ?")]
    async fn find_name(&self, id: i64) -> Result<Option<String>>;
}
```

### Multiple Results
```rust
#[repo]
trait UserRepo {
    // Result<Vec<T>> -> uses fetch_all() - zero or more rows
    #[dml("SELECT name FROM users")]
    async fn all_names(&self) -> Result<Vec<String>>;
}
```

### Tuple Results (From Real Tests)
```rust
#[repo]
trait TupleRepo {
    // Tuple return types are fully supported
    #[dml("SELECT COUNT(id) as count, AVG(age) as avg_age FROM users")]
    async fn average_age(&self) -> Result<(i64, Option<Decimal>)>;

    #[dml("SELECT id, age, name FROM users LIMIT 2")]
    async fn get_all_ages(&self) -> Result<Vec<(i64, u8, String)>>;
}
```

### Stream Results (From Real Tests)
```rust
use futures::{Stream, StreamExt};

#[repo]
trait StreamRepo {
    // fn (not async) returns Stream for memory-efficient processing
    #[dml("SELECT id, name, age FROM users")]
    fn stream_users(&self) -> impl Stream<Item = Result<SimpleUser>> + Send;

    #[dml("SELECT name FROM users ORDER BY name")]
    fn stream_names(&self) -> impl Stream<Item = Result<String>> + Send;
}
```

### Modification Results
```rust
#[repo]
trait UserRepo {
    // Result<QueryResult> -> uses execute() - for INSERT/UPDATE/DELETE
    #[dml("DELETE FROM users WHERE id = ?")]
    async fn delete(&self, id: i64) -> Result<QueryResult>;
}
```

## Parameter Binding

### Positional Parameters
```rust
#[repo]
trait UserRepo {
    // Parameters bound by position: name -> ?, email -> ?, age -> ?
    #[dml("INSERT INTO users (name, email, age) VALUES (?, ?, ?)")]
    async fn create(&self, name: String, email: String, age: u8) -> Result<QueryResult>;
}
```

### Named Parameters
```rust
#[repo]
trait UserRepo {
    // Named parameters - can be in any order in function signature
    #[dml("SELECT * FROM users WHERE name = @name AND age >= @min_age")]
    async fn search(&self, min_age: u8, name: String) -> Result<Vec<User>>;

    // Parameter reuse (PostgreSQL/SQLite only)
    #[dml("SELECT * FROM users WHERE (name = @search OR email = @search) AND age > @min_age")]
    async fn search_user(&self, search: String, min_age: u8) -> Result<Vec<User>>;
}
```

## Compile-Time Safety

SQLx-Data inherits all of SQLx's compile-time verification through the generated `sqlx::query_*!` macros:

### SQL Validation
Your SQL is checked against your actual database at compile time:

```rust
#[repo]
trait UserRepo {
    // ✅ Compiles only if 'users' table and 'name' column exist
    #[dml("SELECT name FROM users WHERE id = ?")]
    async fn get_name(&self, id: i64) -> Result<String>;

    // ❌ Compile error if column doesn't exist
    #[dml("SELECT nonexistent_column FROM users")]
    async fn invalid(&self) -> Result<String>;
}
```

### Type Compatibility
Parameter and return types must match SQL types:

```rust
#[repo]
trait UserRepo {
    // ✅ i64 is compatible with database INTEGER
    #[dml("SELECT name FROM users WHERE id = ?")]
    async fn get_name(&self, id: i64) -> Result<String>;

    // ❌ String is not compatible with INTEGER column
    #[dml("SELECT name FROM users WHERE id = ?")]
    async fn invalid(&self, id: String) -> Result<String>;
}
```

## Error Handling

SQLx-Data uses SQLx's error handling system directly through type aliases:

```rust
// From the actual codebase:
pub type Result<T, E = sqlx::Error> = std::result::Result<T, E>;
```

This means all repository methods use `sqlx::Error`:

```rust
use sqlx_data::Result;  // This is std::result::Result<T, sqlx::Error>

#[repo]
trait UserRepo {
    #[dml("SELECT name FROM users WHERE id = ?")]
    async fn get_name(&self, id: i64) -> Result<String>;
}

// Usage - same error handling as raw SQLx
async fn example(repo: &impl UserRepo) -> Result<()> {
    match repo.get_name(1).await {
        Ok(name) => println!("User: {}", name),
        Err(sqlx::Error::RowNotFound) => println!("User not found"),
        Err(sqlx::Error::Database(db_err)) => println!("Database error: {}", db_err),
        Err(e) => println!("Other error: {}", e),
    }
    Ok(())
}
```

## Zero-Cost Abstractions

SQLx-Data generates code identical to manual SQLx usage:

### Generated Code Performance
The macro generates the same code you would write by hand:

```rust
// Your SQLx-Data code
#[dml("SELECT name FROM users WHERE id = ?")]
async fn get_name(&self, id: i64) -> Result<String>;

// Generates equivalent to manual SQLx:
async fn get_name_query(&self, id: i64) -> Result<String> {
    sqlx::query_scalar!("SELECT name FROM users WHERE id = ?", id)
        .fetch_one(self.get_pool())
        .await
}
```

### Performance Characteristics
- **Runtime overhead**: Zero
- **Binary size**: No increase
- **Compile time**: Small increase for macro processing (~20-90µs per query)

## Override Capability

The two-method pattern allows flexible overriding:

```rust
struct MyApp { pool: Pool }

impl UserRepo for MyApp {
    fn get_pool(&self) -> &Pool { &self.pool }

    // Override with custom logic while reusing generated query
    async fn get_name(&self, id: i64) -> Result<String> {
        // Add logging, caching, validation, etc.
        log::info!("Getting name for user {}", id);

        if let Some(cached) = get_from_cache(id) {
            return Ok(cached);
        }

        let name = self.get_name_query(id).await?;  // Use generated _query method
        cache_name(id, &name);
        Ok(name)
    }
}
```

## Key Principles

### 1. Trait-Based Design
Business logic depends on traits, not implementations. This enables testing and flexibility.

### 2. Signature Replacement, Not Addition
Macros completely replace empty signatures with implementations - no duplication.

### 3. Compile-Time Code Generation
All SQL is verified and transformed at build time using SQLx's proven macros.

### 4. Flexible Override Pattern
The `_query` method pattern allows custom logic while reusing generated SQL code.
